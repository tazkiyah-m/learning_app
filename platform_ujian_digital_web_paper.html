01

Latar Belakang & Tujuan

Pelaksanaan ujian secara konvensional menghadapi berbagai tantangan, mulai dari ketidakefisienan distribusi soal, penilaian yang memakan waktu, hingga kurangnya transparansi umpan balik kepada siswa. Platform Ujian Digital hadir sebagai solusi berbasis web yang dirancang untuk mendigitalisasi seluruh siklus ujian — dari pembuatan soal hingga publikasi nilai.

Sistem ini dibangun sebagai prototipe lokal yang dapat dioperasikan pada lingkungan pengembangan, dengan tujuan membuktikan kelayakan konsep sebelum diperluas ke skala produksi.

Tujuan utama: Membangun platform ujian digital yang memungkinkan guru membuat dan mengelola ujian secara mandiri, siswa mengerjakan ujian dengan antarmuka yang intuitif, serta menyajikan penilaian otomatis dan umpan balik transparan kepada semua pihak.

02

Deskripsi Fitur

Sistem menyediakan dua antarmuka utama yang disesuaikan berdasarkan peran pengguna. Seluruh fitur dapat diakses melalui peramban web tanpa memerlukan pemasangan perangkat lunak tambahan di sisi klien.

🔐

Autentikasi

  • Registrasi akun siswa & guru
  • Login dengan verifikasi peran
  • Password di-hash dengan bcrypt
  • Sesi disimpan via Context API
📝

Dashboard Siswa

  • Daftar ujian yang tersedia & belum dikerjakan
  • Pengerjaan soal pilihan ganda & esai dengan timer
  • Submit dan penilaian otomatis oleh backend
  • Riwayat dengan pembahasan lengkap
🏫

Dashboard Guru

  • Buat ujian (judul, mapel, durasi, soal)
  • Dua tipe soal: pilihan ganda & esai
  • Kelola ujian: lihat detail, hapus
  • Rekap nilai semua siswa
📡

Monitoring Real-Time

  • Siswa mengirim ping setiap 30 detik
  • Guru polling aktif setiap 10 detik
  • Menampilkan daftar siswa yang sedang online
  • Endpoint POST /api/update-activity
03

Arsitektur Sistem

Sistem menggunakan pola arsitektur monolitik berlapis (layered monolith). Backend diorganisasi dalam tiga lapisan yang saling terpisah secara logis namun berjalan dalam satu proses, sehingga memudahkan pengembangan dan debugging pada tahap prototipe.

Gambar 1 — Diagram Arsitektur Logika Platform Ujian Digital

Komunikasi antara frontend dan backend sepenuhnya dilakukan melalui REST API berbasis JSON. Frontend di-render di port 3000, sedangkan backend Golang Gin berjalan di port 8080. Konfigurasi CORS membatasi akses hanya dari origin lokal.

04

Alur Data Fitur Utama

Berikut adalah alur data untuk setiap fitur utama, yang menggambarkan interaksi antara lapisan frontend, backend, dan database.

4.1 Registrasi & Login

LangkahKomponenProses
1Frontend (Form)Pengguna mengisi formulir → Next.js mengirim POST /api/register atau POST /api/login
2Handler (Layer 1)Menerima request JSON, memvalidasi input, meneruskan ke Service
3Service (Layer 2)Registrasi: hash password dengan bcrypt → simpan ke DB. Login: bcrypt.CompareHashAndPassword, cek role
4Repository (Layer 3)Eksekusi query SQL INSERT atau SELECT ke tabel users
5Frontend (Context)Login sukses → data user disimpan di UserContext → redirect ke dashboard sesuai role

4.2 Pengerjaan Ujian (Siswa)

LangkahKomponenProses
1FrontendGET /api/siswa/ujian-tersedia → tampilkan daftar ujian yang belum dikerjakan
2FrontendSiswa klik "Mulai" → GET /api/siswa/ujian/:id → ambil soal-soal
3FrontendSiswa menjawab soal satu per satu; jawaban dibandingkan di frontend untuk soal pilihan ganda
4FrontendSelesai → POST /api/siswa/submit-jawaban dengan payload jawaban lengkap
5Service (Layer 2)Hitung nilai otomatis, simpan hasil ke tabel hasil_ujian dengan kolom detail_jawaban (JSON)
6FrontendTampilkan nilai, redirect ke dashboard siswa

4.3 Pembuatan Ujian (Guru)

LangkahKomponenProses
1Frontend (Form)Guru mengisi judul, mata pelajaran, durasi, dan soal-soal
2FrontendPOST /api/ujian dengan payload lengkap ujian + array soal
3Service (Layer 2)Validasi data, buat record di tabel ujian dan soal
4Repository (Layer 3)INSERT ke tabel ujian, kemudian loop INSERT soal satu per satu
5FrontendRespons sukses → ujian langsung tersedia untuk siswa

4.4 Monitoring Siswa Online

LangkahIntervalProses
Ping SiswaSetiap 30 detikSaat siswa login dan membuka dashboard, frontend mengirim POST /api/update-activity secara periodik; backend memperbarui kolom last_active di tabel users
Polling GuruSetiap 10 detikDashboard guru memanggil GET /api/active-students; backend mengambil siswa dengan last_active dalam 60 detik terakhir

06

Keamanan Sistem

Sebagai prototipe lokal, sistem menerapkan lapisan keamanan dasar yang cukup untuk melindungi data pengguna dan mencegah akses tidak sah dalam skenario demonstrasi.

Aspek Implementasi Detail Teknis Status
Hashing Password bcrypt dengan cost factor default (10) Golang: golang.org/x/crypto/bcrypt. Password plain-text tidak pernah disimpan di database. Diterapkan
Manajemen Sesi Context API + localStorage (Next.js) Data pengguna (id, role, token) disimpan di UserContext yang di-persist via localStorage. Sesi hilang saat browser ditutup. Demo saja
CORS Dibatasi hanya origin lokal Middleware Gin: cors.AllowOrigins(["http://localhost:3000"]). Mencegah cross-origin request dari domain lain. Diterapkan
Otorisasi Peran Pengecekan role di setiap handler Setiap endpoint sensitif (guru/siswa) memverifikasi role dari payload sesi sebelum memproses request. Diterapkan
JWT / Token Belum diimplementasikan Autentikasi berbasis token stateless (JWT) belum diterapkan. Direkomendasikan untuk versi produksi. Belum ada
HTTPS / TLS Tidak diperlukan (lokal) Koneksi berjalan melalui HTTP di localhost. Untuk deployment, wajib menggunakan TLS/HTTPS. Lokal saja
07

Keunggulan & Keterbatasan

Sebagai prototipe demo lokal, sistem ini memiliki keunggulan yang menonjol sekaligus keterbatasan yang perlu ditangani sebelum digunakan dalam lingkungan produksi.

Keunggulan

  • Arsitektur tiga lapisan memisahkan tanggung jawab dengan jelas, mempermudah pemeliharaan kode
  • Penilaian ujian dilakukan otomatis oleh backend, mengurangi beban guru
  • Riwayat ujian lengkap dengan pembahasan membantu proses belajar siswa secara mandiri
  • Monitoring real-time memungkinkan guru memantau kehadiran siswa tanpa alat tambahan
  • Stack teknologi modern (Next.js + Golang) memberikan performa tinggi dan DX yang baik
  • Password di-hash dengan bcrypt, memenuhi standar keamanan dasar

Keterbatasan

  • Tidak ada JWT/token stateless — sesi rentan terhadap manipulasi di lingkungan tidak aman
  • Monitoring menggunakan polling (bukan WebSocket), kurang efisien untuk skala besar
  • CORS hanya untuk localhost — perlu konfigurasi ulang untuk deployment multi-host
  • Belum ada mekanisme refresh token atau manajemen sesi yang kedaluwarsa
  • Fitur hapus ujian yang sudah ada hasil belum dilindungi dengan validasi integritas data
  • Terbatas untuk lingkungan lokal; belum siap untuk deployment cloud tanpa refactoring keamanan

Rekomendasi pengembangan lanjutan: Implementasi JWT untuk autentikasi stateless, migrasi polling ke WebSocket untuk monitoring efisien, penambahan rate limiting pada endpoint login, serta migrasi ke Docker Compose untuk kemudahan deployment.
